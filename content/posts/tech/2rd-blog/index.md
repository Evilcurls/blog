起因是在网上搜MedicalGPT的时候看到了一个哥们使用DoIT主题贼炫酷，然后赶紧clone code让Antigravity给我原来的移植，没想到做的很成功但是部署到Github Pages上难点重重。我打算先写我学到的东西，然后再讲困难。



### 图床

​	在之前写博客的时候，图片都是需要我先放到文件夹里然后我使用相对路径导入，这样繁琐的步骤消磨之下，我就投入到了富文本编辑器的怀抱，然后仔细阅读其他人的博客步骤我发现，他们是将图片保存到云端，然后在Markdown里复制云端oss提供的链接完成的，听起来好像跟原来一样麻烦，但实际省时间的步骤是不用记录相对路径，而且有Picgo这样的软件一步到位，我使用了Github作为图床，只需要一个access token，picgo就能上传文件到那并返回路径。

​	后记：2026年5月27日，飞书导出的markdown文件，如果原来在飞书的图片则自动生成 一个链接，操，白学了，但是说不定我的一些资料不想让飞书看呢，嘻嘻。

### Github Action自动部署

​	由于我用的都是渲染器，需要在本地运行 Hugo Server -D才能渲染出网页，而Github Pages又要求必须是静态网页比如Html这样的，我始终秉承着自动化原则，希望自己写完文档就可以出网页，所以一次次渲染出html再git push不是我所期待的，然后Github Action就相当于可以自动在云端执行我那个Hugo Server -D的操作，不过也正是这个选择造成了我忙碌了一下午的困难。

![](https://raw.githubusercontent.com/Evilcurls/image4blog/main/20260530160025611.png)

### 困难们

​	1.Picgo 我一开始下错了版本，我操了，重复安装了五六遍都不行，安装文件夹里只有一个unistall.exe,后来我才发现我安装的是arm64版本，应该下x64版本

​	2.网页中主页各个博客显示混乱，后来我发现（1）如果是一级标题，就会显示特别大，我尽量避免使用一级标题 （2）有些时候主页的文章甚至预览会很长很长，我发现如果是开头是连续东西（结构图 代码）就会显示很长，但是是正文的话多写点字就没了

​	3.部署到Github Action上 困难重重： 
​		（1） 一开始的DoIT主题是快捷方式，我实际还要下载真实的doit主题
​		（2） hugo版本一会新一会旧，dolt用的try 语法 hugo 1.1太老支持不了，hugo 后面的版本，dolt又用了老版本hugo的数据命名格式，真费劲
​		（3）上传文件在Dolt命名的时候也很麻烦，因为有一个_partials文件夹，在github上运行的时候一定要partials文件夹

  		以上只是部署的问题，我眼睛一直顶着部署时候的倒数，结果好不容易部署上去，显示那才是道阻且长

​	4.部署完了显示问题

​				不显示我资源/链接到其他地方乱码/第二页又全消失了/post没东西，还好这些都使用coding agent弄完了

## 感想

​	我自己在改代码上遵循的路径是，看着别人写的报错，然后自己一步步找，有可能要考虑多个文件，但也仅仅是找，我觉得这是一个机械化的流程，反而是Ai最擅长的点，我以后应该多拥抱AI哪怕程序员会不会死。
​	再往上说一层，写代码其实是中介的活，把A写的主题 B的数据库 C的校验程序粘合在一块，这样工程学习的本质不是提高，而是熟悉别人写的代码，未来的ai可能就是要干掉这些磨合的成本，让大家的初心靠在一起，消灭了心之壁。
